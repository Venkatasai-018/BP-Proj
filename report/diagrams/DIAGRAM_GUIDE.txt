2. REAL-TIME BUS TRACKING FLOW DIAGRAM
   File: 2_bus_tracking_flow.png
   Type: Process Flow Diagram (Three-column layout)
   
   Description:
   This comprehensive diagram shows the real-time GPS tracking workflow from
   three perspectives: Student App, Backend API, and Driver App. It demonstrates
   how location data flows through the system, from the driver's GPS updates
   to the student's real-time bus location display.
   
   Key Components:
   STUDENT APP COLUMN:
   - Opening dashboard and viewing assigned bus
   - Requesting live location from backend
   - Displaying bus location on interactive map
   - Showing estimated time of arrival (ETA)
   - Auto-refresh loop (10-30 seconds)
   
   BACKEND API COLUMN:
   - JWT token verification for security
   - Querying latest bus location from database
   - Calculating ETA using Haversine algorithm
   - Returning location data with timestamp
   
   DRIVER APP COLUMN:
   - Starting trip and enabling GPS tracking
   - Capturing GPS coordinates continuously
   - Sending location updates every 10 seconds
   - Trip active loop with decision diamond
   - End trip functionality
   
   Use in Report:
   Essential for Chapter 5 (Implementation) to explain the core tracking feature.
   Also valuable in Chapter 3 (Requirements) to show functional requirements.

6. DATABASE SCHEMA DIAGRAM
   File: 6_database_schema.png
   Type: Entity-Relationship Diagram (ERD)
   
   Description:
   This diagram presents the complete database schema showing all entities,
   their attributes, primary keys (PK), foreign keys (FK), and relationships
   between tables. It visualizes the relational database structure.
   
   Key Entities and Attributes:
   
   ADMIN TABLE:
   - id (PK)
   - username
   - password_hash
   - email
   - phone
   
   STUDENT TABLE:
   - id (PK)
   - roll_number
   - name
   - email
   - phone
   - route_id (FK ‚Üí Route)
   
   DRIVER TABLE:
   - id (PK)
   - name
   - license_no
   - phone
   - email
   
   BUS TABLE:
   - id (PK)
   - bus_number
   - capacity
   - route_id (FK ‚Üí Route)
   - driver_id (FK ‚Üí Driver)
   
   ROUTE TABLE:
   - id (PK)
   - name
   - route_number
   - start_location
   - end_location
   
   SCHEDULE TABLE:
   - id (PK)
   - bus_id (FK ‚Üí Bus)
   - route_id (FK ‚Üí Route)
   - start_time
   - end_time
   
   FEEDBACK TABLE:
   - id (PK)
   - user_id (FK)
   - category
   - rating
   - comment
   
   BUSLOCATION TABLE:
   - id (PK)
   - bus_id (FK ‚Üí Bus)
   - latitude
   - longitude
   - timestamp
   
   ROUTESTOP TABLE:
   - id (PK)
   - route_id (FK ‚Üí Route)
   - stop_name
   - latitude
   - longitude
   
   Relationships:
   - Route ‚Üí Student (1:N) - One route has many students
   - Route ‚Üí Bus (1:N) - One route can have multiple buses
   - Driver ‚Üí Bus (1:N) - One driver can drive multiple buses
   - Bus ‚Üí Schedule (1:N) - One bus has multiple schedules
   - Route ‚Üí Schedule (1:N) - One route has multiple schedules
   - Bus ‚Üí BusLocation (1:N) - One bus has multiple location records
   - Route ‚Üí RouteStop (1:N) - One route has multiple stops
   
   Use in Report:
   Essential for Chapter 3 (Requirements) or Chapter 4 (System Design) to
   show database design. Demonstrates normalization and data integrity
   through foreign key relationships.

TROUBLESHOOTING:

If diagrams appear blurry in Word:
1. Right-click image ‚Üí Format Picture
2. Picture tab ‚Üí Compression
3. Uncheck "Compress images in file"
4. Select "Best quality for printing (220 ppi)"

If colors don't match when printed:
- Ensure printer is set to "Color" mode
- Use "Best Quality" print setting
- Consider RGB to CMYK conversion for professional printing

If diagrams are too large:
- Resize proportionally (hold Shift while dragging corners)
- Recommended width: 6-7 inches for portrait orientation
- Recommended width: 9-10 inches for landscape orientation
